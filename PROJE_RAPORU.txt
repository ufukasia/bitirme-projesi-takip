SECTION 1: BUG FINDING & FIXING
===========================================================================

Before writing any new code, we analyzed the original app.py carefully to
identify all the problems. Here is what we found and how we fixed each one.

---------------------------------------------------------------------------
BUG 1 — HARDCODED PASSWORD (Security Risk)
---------------------------------------------------------------------------
WHAT WAS WRONG:
  The default password "12345" was written directly inside the source code:
    DEFAULT_PASSWORD = "12345"
  This means anyone who looks at the code (e.g., on GitHub) can see the
  default password immediately. This is a serious security vulnerability.

WHY IT MATTERS:
  If the code is ever shared publicly or committed to a git repository,
  the password is exposed. Even in a student project, this is bad practice.

HOW WE FIXED IT:
  We moved the password to a separate file called ".env" (environment file).
  The code now reads the password from this file:
    DEFAULT_PASSWORD = os.getenv("DEFAULT_PASSWORD", "12345")
  The .env file is listed in .gitignore, so it is NEVER uploaded to GitHub.
  We also created .env.example as a template for others to follow.

---------------------------------------------------------------------------
BUG 2 — THREAD SAFETY (Data Corruption Risk)
---------------------------------------------------------------------------
WHAT WAS WRONG:
  Streamlit can handle multiple users at the same time. When two users try
  to write to the SQLite database simultaneously, the data could get
  corrupted or one write operation could overwrite the other's.

  Example: Student A and Student B both submit their task updates at
  exactly the same millisecond → the database gets confused and one
  update is lost.

HOW WE FIXED IT:
  We added a "threading lock" — think of it like a bathroom door lock.
  Only one person can enter at a time. While one user is writing to the
  database, all other write requests wait their turn.
  
  Code change in db.py:
    db_lock = threading.Lock()
    with db_lock:
        conn.execute(query, params)
        conn.commit()

---------------------------------------------------------------------------
BUG 3 — INSECURE FILE UPLOADS
---------------------------------------------------------------------------
WHAT WAS WRONG:
  Students can upload evidence files (PDFs, images). The original code
  accepted ANY file type without any checks. A malicious user could
  upload dangerous files like .exe or .sh scripts.

HOW WE FIXED IT:
  We added an allowed file types list. The upload widget only accepts:
    ["png", "jpg", "jpeg", "gif", "webp", "pdf", "docx", "zip"]
  Any other file type is rejected automatically by Streamlit.
  Files are stored in /uploads/ folder which is excluded from git.

---------------------------------------------------------------------------
BUG 4 — CODE DUPLICATION (Maintainability Problem)
---------------------------------------------------------------------------
WHAT WAS WRONG:
  The same database queries and UI rendering code was copy-pasted in many
  different places throughout app.py. For example, the code to display a
  student's tasks was written 3 separate times with slight differences.

WHY IT MATTERS:
  If you need to fix a bug in that code, you have to find and fix it in
  3 different places. If you miss one, the bug remains. This makes
  maintenance very difficult.

HOW WE FIXED IT:
  We created reusable functions in components.py and ui_helpers.py.
  Now the code exists in ONE place and is called from everywhere else.
  If we fix a bug or improve the design, it applies everywhere instantly.

---------------------------------------------------------------------------
BUG 5 — UNNECESSARY SYNC ON EVERY RERUN (Performance)
---------------------------------------------------------------------------
WHAT WAS WRONG:
  Every time ANY user interaction happened (clicking a button, typing
  in a text box), Streamlit reruns the entire script. The sync_auth_users()
  function was being called every single time, performing heavy database
  operations on every rerun.

  This made the app feel slow and unresponsive.

HOW WE FIXED IT:
  We use Streamlit's session_state to track whether sync has already
  been done for this user session:
    if not st.session_state.get("_sync_done"):
        sync_auth_users(conn)
        st.session_state["_sync_done"] = True
  Now sync runs ONCE per session, not hundreds of times.

---------------------------------------------------------------------------
BUG 6 — STATUS TRANSITION ERRORS
---------------------------------------------------------------------------
WHAT WAS WRONG:
  The task status dropdown was showing all statuses (TODO, DOING, DONE)
  to every user at all times. A student could mark a task as DONE
  without ever setting it to DOING first. The milestone ordering logic
  was also incorrect.

HOW WE FIXED IT:
  We created an allowed_status_options() function in utils.py that
  returns only the statuses a task is allowed to transition to:
    - TODO task can only go to: DOING
    - DOING task can go to: DOING or DONE
    - DONE task stays DONE (advisors can override)


===========================================================================
SECTION 2: CODE ARCHITECTURE — SPLITTING THE MONOLITH
===========================================================================

WHAT WAS WRONG BEFORE:
  The entire application was in a single file: app.py
  It had approximately 1,200 lines of code doing everything:
  - Database connections
  - Business logic
  - UI rendering
  - Authentication
  - File uploads
  - All three panels (advisor, leader, student)
  
  This is called a "monolithic" architecture. It's like having one giant
  room in a house that serves as kitchen, bedroom, bathroom, and office 
  all at once — very hard to work in.

WHAT WE DID — THE REFACTORING:
  We applied the "Single Responsibility Principle" (SRP): every file
  should do ONE thing and do it well.

Here is every new file we created and exactly what it contains:

---------------------------------------------------------------------------
FILE: constants.py
---------------------------------------------------------------------------
PURPOSE: Store all fixed values that don't change during runtime.

WHAT'S IN IT:
  - DB_PATH: Path to the SQLite database file
  - DEFAULT_PASSWORD: Loaded from .env file
  - MIN_PASSWORD_LEN: Minimum password length requirement
  - MILESTONE_LABELS: Dictionary mapping milestone keys to display names
    e.g., {"M1": "Literatür Taraması", "M2": "Algoritma Planı", ...}
  - MILESTONES: List of (key, label) tuples for all 6 milestones
  - PRIORITY_OPTIONS: ["Dusuk", "Orta", "Yuksek"]
  - ROLE_OPTIONS: List of available team roles
  - LANGUAGE_OPTIONS: {"🇹🇷 Türkçe": "tr", "🇬🇧 English": "en"}

WHY IT MATTERS:
  If we need to change the milestone names or add a new priority level,
  we change it in ONE place and it applies everywhere automatically.

---------------------------------------------------------------------------
FILE: utils.py  
---------------------------------------------------------------------------
PURPOSE: Small helper functions used across multiple modules.

KEY FUNCTIONS:
  - hash_password(password): Converts "12345" into a scrambled hash
    string using HMAC-SHA256. The original password cannot be recovered
    from the hash. Stored hash: "ab4f3c9d1e..."
    
  - verify_password(stored_hash, entered_password): Checks if the hash
    of what the user typed matches the stored hash. Returns True/False.
    
  - status_tr(status_code): Converts "TODO" → "Yapılacak", "DOING" →
    "Devam Ediyor", "DONE" → "Tamamlandı". Used for display.
    
  - allowed_status_options(current_status): Returns the list of statuses
    that a task is allowed to change to from its current state.
    
  - normalize_str(text): Removes Turkish special characters for safe
    file naming (ş→s, ğ→g, ı→i, etc.)

---------------------------------------------------------------------------
FILE: db.py
---------------------------------------------------------------------------
PURPOSE: Everything related to the database connection and schema.

KEY PARTS:
  - get_conn(): Opens a connection to the SQLite database file. Uses
    @st.cache_resource so the connection is opened only ONCE and reused.
    
  - init_schema(conn): Creates all database tables if they don't exist:
    * students: (student_no, student_name, project_name, advisor_name, ...)
    * auth_users: (user_id, role, password_hash, must_change_password, ...)
    * tasks: (id, project_name, milestone_key, title, status, priority, ...)
    * task_comments: (id, task_id, author_id, author_role, comment, ...)
    * weekly_updates: (id, project_name, student_no, task_id, ...)
    * feedbacks: (id, project_name, advisor_name, feedback, ...)
    * leaders: (project_name, student_no)
    * member_roles: (project_name, student_no, role, responsibility)
    
  - fetch_df(conn, query, params): Runs a SELECT query and returns a
    pandas DataFrame. All columns are accessible by name.
    
  - db_lock: A threading.Lock() object to prevent concurrent write
    corruption.

---------------------------------------------------------------------------
FILE: models.py
---------------------------------------------------------------------------
PURPOSE: All database business logic — reading and writing data.
         This is the "brain" of the application.

KEY FUNCTIONS (examples):
  - authenticate_user(conn, user_id, password, role): Checks credentials.
    Returns the user record if correct, or None if wrong.
    
  - fetch_tasks(conn, project_name): Gets all tasks for a project as a
    DataFrame. Used by all three panels.
    
  - create_task(conn, project_name, milestone_key, title, ...): Inserts a
    new task record. Called only by the group leader.
    
  - update_task(conn, task_id, new_status, evidence_link, ...): Updates a
    task. Enforces milestone ordering rules (cannot skip milestones).
    
  - build_project_metrics(conn, roster, projects): Calculates completion %,
    overdue count, and risk level for each project. Used in the advisor
    panel's project overview cards.
    
  - member_progress(project_members, tasks_df): For each student, counts
    their total tasks, completed tasks, and calculates their %.
    
  - get_leader(conn, project_name): Returns the student_no of the group
    leader for a project.
    
  - sync_auth_users(conn): Makes sure every student in the students table
    has a corresponding login account in auth_users.

---------------------------------------------------------------------------
FILE: components.py
---------------------------------------------------------------------------
PURPOSE: Reusable UI pieces that appear in multiple panels.

KEY FUNCTIONS:
  - render_task_comments(conn, task_id, ...): Displays the comment thread
    for a task with colored bubble cards by role, plus a form to add a
    new comment.
    
  - render_milestone_progress(tasks_df): Displays the 6-column milestone
    grid with colored mini-cards showing % completion per milestone.
    
  - render_evidence_file(file_path): Displays an uploaded file (image
    directly shown, PDF/other shown as download link).
    
  - save_uploaded_evidence(uploaded_file, task_id): Saves an uploaded
    file to /uploads/ folder with a unique name.

---------------------------------------------------------------------------
FILE: ui_helpers.py
---------------------------------------------------------------------------
PURPOSE: Advanced HTML-based visual components. Returns styled HTML
         that Streamlit renders with st.markdown(unsafe_allow_html=True).

KEY FUNCTIONS:
  - section_header(icon, title, subtitle): Creates a styled section
    divider with an icon, bold title, and optional subtitle.
    Example: section_header("📊", "Milestone Progress", "By project")
    
  - status_badge_html(status): Returns an HTML pill badge:
    TODO → grey pill, DOING → blue pill, DONE → green pill.
    
  - priority_badge_html(priority): Low=green, Medium=yellow, High=red.
  
  - risk_badge_html(risk): 🟢 Low, 🟡 Medium, 🔴 High with colors.
  
  - render_project_cards(projects_data): Renders a responsive 3-column
    grid of project summary cards, each showing project name, leader,
    member count, progress bar, risk badge, overdue count.
    
  - render_active_task_card(task_row, milestone_label): Renders a
    highlighted gradient card showing the currently active task with
    its title, status/priority badges, and deadline.
    
  - render_feedback_card(fb_row): Renders an advisor feedback with a
    colored left border (red=revision needed, blue=normal feedback).
    
  - render_member_table(project_members, roles_df, leader_no): Renders
    an HTML table of team members with colored role badges.

  IMPORTANT DETAIL — _e() and _t() functions:
    _e(text): Escapes HTML special characters in user content (< > & " ')
              to prevent HTML injection bugs from database content.
    _t(text): Translates a Turkish string to English if English mode is on.

---------------------------------------------------------------------------
FILE: styles.py
---------------------------------------------------------------------------
PURPOSE: One big CSS string injected into Streamlit to override its
         default appearance with our professional dark navy/gold theme.

HOW IT WORKS:
  Streamlit renders HTML in the browser like any website. We inject custom
  CSS using st.markdown("<style>...</style>", unsafe_allow_html=True).
  This CSS overrides Streamlit's default white/light-blue appearance.

KEY THEME CHOICES:
  - Background: #0a2342 (dark navy) for the sidebar
  - Primary color: #ffd700 (gold/yellow) for highlights and badges
  - Accent: #0f4c81 (medium navy) for cards and borders
  - Font: Inter from Google Fonts (clean, modern, professional)
  - Cards: White background, subtle shadow, border-radius: 12px
  - Buttons: Gradient fill (#0f4c81 → #1a3a6b), hover lift animation
  - Tables: Navy header row, alternating row colors, hover highlight

---------------------------------------------------------------------------
FILE: i18n.py  (Internationalization)
---------------------------------------------------------------------------
PURPOSE: Makes the entire application available in both Turkish and English.

HOW IT WORKS (Technical Detail):
  Streamlit renders buttons, labels, and text via functions like
  st.button("Kaydet"), st.text_input("Öğrenci No"), etc.
  
  We use a technique called "monkey-patching": we replace Streamlit's
  original functions with our own wrapper functions that translate the
  text before passing it to Streamlit.
  
  Example (simplified):
    original_button = st.button  
    def translated_button(label, *args, **kwargs):
        if language == "en":
            label = translate(label)  # "Kaydet" → "Save"
        return original_button(label, *args, **kwargs)
    st.button = translated_button
  
  Now every call to st.button() in the entire app is automatically
  translated without changing any other code!

TRANSLATION SOURCES (in order of priority):
  1. TRANSLATION_OVERRIDES_TR_EN: ~130 exact phrase matches
     e.g., "Görev Yorumları" → "Task Comments"
     
  2. SEGMENT_TRANSLATIONS_TR_EN: Substring replacements for longer text
     e.g., "Danisman paneli" → "Advisor panel" inside longer strings
     
  3. TOKEN_TRANSLATIONS_TR_EN: Word-by-word translation for known words
     e.g., "Gorev" → "Task", "Ogrenci" → "Student"
     
  4. TEXT_CLEANUPS_EN: Fix awkward auto-translations after the above

IMPORTANT SAFETY RULE:
  The translator detects if a string contains HTML tags (<div>, </div>)
  and skips those strings entirely to avoid breaking the layout.
  Instead, for HTML blocks we call _t() manually on each label.

---------------------------------------------------------------------------
FILE: panels/advisor.py, panels/leader.py, panels/student.py
---------------------------------------------------------------------------
PURPOSE: Each file contains the complete UI for one user role.

  advisor.py: Project card grid, student search, leader assignment,
              CSV upload, password reset, project task management,
              feedback history, weekly updates per project.
              
  leader.py:  Hero banner, milestone progress, team member table,
              role assignment form, task creation form (2-column),
              task tracking table, task update form, comments,
              member progress summary, password reset for members.
              
  student.py: Profile hero banner, project metrics, milestone grid,
              team member table, personal task queue table,
              active task card + update form, weekly report form,
              past weekly updates, advisor feedback, group comparison.

SECTION 3: CONFIGURATION MANAGEMENT (.env FILE)
===========================================================================

WHAT IS A .env FILE?
  A plain text file that stores configuration values for the application.
  Instead of writing settings directly in the code, you read them from
  this file. This separates "what the app needs" from "what values to use."

THE PROBLEM IT SOLVES:
  Without .env, if we have:
    DB_PATH = "/Users/khaled/project/project_tracker.db"
  This path works only on Khaled's computer. On the professor's computer,
  the path is completely different and the app would crash.
  
  With .env, each person has their own .env file with their own path, and
  the code reads whatever is set there.

OUR .env FILE CONTENTS:
  DB_PATH=project_tracker.db          ← Where to store the database
  DEFAULT_PASSWORD=12345              ← Starting password for new users
  MIN_PASSWORD_LEN=5                  ← Minimum characters for new passwords
  ADMIN_ADVISOR_KEYS=dr.mehmet        ← Who has admin access
  DEFAULT_ADVISOR=dr.mehmet           ← Default advisor name
  UPLOADS_DIR=uploads                 ← Folder for uploaded files
  DEFAULT_LANGUAGE=tr                 ← Default UI language

HOW constants.py READS IT:
  from dotenv import load_dotenv
  load_dotenv()  ← Reads .env file into environment variables
  DB_PATH = os.getenv("DB_PATH", "project_tracker.db")
  
  The second argument ("project_tracker.db") is the fallback if the
  variable is not set in .env — so the app still works even without .env.

.env vs .env.example:
  .env         → Your actual secret values. NEVER commit to git.
  .env.example → A template with placeholder values showing what keys exist.
                 Safe to commit to git and share with others.

SECTION 4: VIRTUAL ENVIRONMENT (.venv)
===========================================================================

WHAT IS A VIRTUAL ENVIRONMENT?
  A virtual environment is an isolated Python installation just for your
  project. It has its own copy of Python and all the packages you install,
  completely separate from Python installed on the system.

WHY DO WE NEED IT?
  Imagine you have two projects:
  - Project A needs streamlit version 1.40
  - Project B needs streamlit version 1.55
  
  Without virtual environments, you can only have ONE version installed
  globally. They would conflict with each other.
  
  With virtual environments, each project has its own isolated copy.
  Changing packages for one project never affects the other.

HOW TO CREATE AND USE IT:
  Step 1 — Create it (only once):
    python3 -m venv .venv
    This creates a hidden folder ".venv/" containing a full Python setup.
  
  Step 2 — Activate it (every time you open a new terminal):
    source .venv/bin/activate        (Mac/Linux)
    .venv\Scripts\activate           (Windows)
    Your terminal prompt changes to: (.venv) $
  
  Step 3 — Install packages:
    pip install -r requirements.txt
    This installs streamlit, pandas, and python-dotenv into .venv only.
  
  Step 4 — Run the app (while activated):
    streamlit run app.py

WHY .venv IS NOT UPLOADED TO GIT:
  .venv/ folder is about 200MB and contains thousands of files.
  It is machine-specific (the paths inside are hardcoded to your computer).
  Anyone else can recreate it in 30 seconds with "pip install -r requirements.txt".
  We added .venv/ to .gitignore to exclude it from version control.

SECTION 6: MULTI-LANGUAGE SUPPORT
===========================================================================

The application supports both Turkish and English. The language selector
appears in the left sidebar (radio buttons: 🇹🇷 Türkçe / 🇬🇧 English).
When switched, the entire page reruns in the new language.

HOW THE MONKEY-PATCHING WORKS IN DETAIL:

  patch_streamlit_i18n() is called once at app startup.
  It loops through these Streamlit functions:
    ["title", "header", "subheader", "caption", "markdown", "text",
     "success", "error", "warning", "info", "button", "checkbox",
     "text_input", "text_area", "file_uploader", "date_input",
     "form_submit_button", "download_button", "expander"]
  
  For each function, it creates a wrapper that:
    1. Takes the label (first argument)
    2. Passes it through translate_text()
    3. Calls the original Streamlit function with the translated label
  
  For selectbox/radio/multiselect, it also handles the format_func

SECTION 7: SECURITY IMPROVEMENTS
===========================================================================

---------------------------------------------------------------------------
7a. PASSWORD HASHING
---------------------------------------------------------------------------
  Passwords are NEVER stored as plain text. When a user sets "newpass123":
    1. We combine the password with a secret salt: "newpass123" + SALT
    2. We run HMAC-SHA256 on it → produces a 64-character hash string
    3. We store the hash in the database: "3d8f9bc2a1..."
  
  When the user logs in with "newpass123":
    1. We compute the hash of what they typed
    2. We compare it to the stored hash using hmac.compare_digest()
       (constant-time comparison to prevent timing attacks)
    3. If identical → login succeeds. The original password is never needed.
  
  Even if someone steals the database file, they cannot recover the passwords.

---------------------------------------------------------------------------
7b. FORCED PASSWORD CHANGE ON FIRST LOGIN
---------------------------------------------------------------------------
  New users have DEFAULT_PASSWORD as their password AND a flag:
    must_change_password = 1 (True)
  
  When such a user logs in, instead of showing the normal panel, we show
  a password change form. They CANNOT skip it. After changing, the flag
  is set to 0 and they see the normal panel.
  
  This ensures no one continues using the insecure default password.

---------------------------------------------------------------------------
7c. ROLE-BASED ACCESS CONTROL
---------------------------------------------------------------------------
  The routing logic in app.py strictly enforces what each role can see:
  
  ADVISOR (role="advisor"):
    → Always goes to render_advisor_panel()
    → Can see all projects they advise
    → Admin advisors can reset the database
  
  LEADER (is_leader=True, role="student"):
    → Goes directly to render_leader_panel()
    → Can only manage their own project
    → Cannot access advisor functions
    → There is NO radio button to switch to student view
  
  STUDENT (is_leader=False, role="student"):
    → Goes directly to render_student_panel()
    → Can ONLY see their own tasks
    → No path to reach the leader panel exists in the code
  
  The old code had a radio button "Öğrenci / Grup Yöneticisi" which allowed
  leaders to toggle between views. We REMOVED this toggle entirely.
  Role separation is now strict and enforced.

---------------------------------------------------------------------------
7d. XSS PROTECTION IN HTML OUTPUT
---------------------------------------------------------------------------
  XSS = Cross-Site Scripting. It's when user input contains HTML or
  JavaScript that gets rendered and executed in the browser.
  
  Example attack: A student sets their task title to:
    <script>alert('hacked!')</script>
  
  If we put this directly in our HTML template:
    f"<div>{task_title}</div>"
  The browser executes the script!
  
  Our fix: All user content passes through _e() = html.escape()
    _e("<script>alert('hacked!')</script>")
    → "&lt;script&gt;alert('hacked!')&lt;/script&gt;"
  
  This is rendered as literal text, not as HTML/JavaScript. Safe!

---------------------------------------------------------------------------
7e. GIT PROTECTION (.gitignore)
---------------------------------------------------------------------------
  We added these rules to .gitignore so they are never uploaded to GitHub:
  
    .env              ← Contains passwords and secrets
    .venv/            ← Large machine-specific virtual environment
    uploads/          ← User-uploaded files (privacy)
    *.db              ← Database contains all student/project data
    *.db-shm          ← SQLite temporary file
    *.db-wal          ← SQLite write-ahead log

SECTION 9: HOW THE APP FLOWS (End-to-End Walkthrough)
===========================================================================

  1. User runs: streamlit run app.py
  
  2. app.py starts:
     a. Sets page config (title, icon, layout="wide")
     b. Calls inject_styles() → applies the navy/gold CSS theme
     c. Calls patch_streamlit_i18n() → wraps all Streamlit functions
     d. Opens DB connection via get_conn() (cached, only opens once)
     e. Calls init_schema() → creates tables if they don't exist
     f. Loads all students from DB into all_roster DataFrame
     g. On first run per session: bootstrap_defaults(), sync_auth_users()
  
  3. Login flow:
     a. If no user in session_state → renders centered login card
     b. User picks role (Student/Advisor), enters ID + password
     c. authenticate_user() checks HMAC hash against stored hash
     d. If match: stores user dict in st.session_state["auth_user"]
     e. Streamlit reruns → user is now logged in
  
  4. If just logged in with DEFAULT_PASSWORD:
     enforce_password_change() shows mandatory password form
     User changes password → must_change_password set to 0 → reruns
  
  5. Routing (in app.py after auth):
     if role == "advisor":
         → get_roster_from_db() for advisor's students
         → render sidebar with advisor info
         → render_advisor_panel()
     
     elif is_leader (student who is group leader):
         → render_leader_panel()
     
     else (regular student):
         → render_student_panel()
  
  6. Inside each panel:
     - Reads data from DB via models.py functions
     - Renders HTML hero banner
     - Calls section_header() for each section
     - Shows forms, tables, comments using Streamlit + ui_helpers.py
     - All button labels, form labels etc. auto-translated via i18n patches
     - User submits a form → model function updates DB → st.rerun()
